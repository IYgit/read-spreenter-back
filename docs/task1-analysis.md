## Проблема 1 — `GET /api/auth/me` повертає 401 у Swagger

### Причина

У `JwtUtil` є **подвійне кодування** секретного ключа:

```java
// JwtUtil.java — конструктор
byte[] keyBytes = Decoders.BASE64.decode(
    java.util.Base64.getEncoder().encodeToString(secret.getBytes()) // ← BASE64 encode, потім decode
);
```

Виходить: беремо рядок → кодуємо в Base64 → декодуємо з Base64 → отримуємо **оригінальний рядок у байтах**.
Це не помилка само по собі, але ключ `"read-sprinter-super-secret-key-change-in-production-min-256-bits"` — **це звичайний рядок**, не Base64-рядок. Після цієї операції він скорочується.

Перевірмо: цей ключ в UTF-8 = 64 байти → після подвійного перекодування залишається коротший масив байт. JJWT може тихо прийняти такий ключ при генерації, але відхилити при валідації — залежно від внутрішньої реалізації. Саме це спричиняє `JwtException` у `validateToken`, фільтр не встановлює `Authentication` в `SecurityContext`, і `/api/auth/me` повертає 401.

**Також є ще одна причина**: `/api/auth/**` у `SecurityConfig` дозволено для всіх (`permitAll()`), але `GET /api/auth/me` — потребує авторизованого користувача. Оскільки цей шлях підпадає під `permitAll()`, Spring не вимагає токен, проте й не встановлює `Authentication` якщо токен невалідний → `authentication.isAuthenticated()` = false → контролер сам повертає 401.

Таким чином є **дві незалежні причини**:
1. Ключ JWT генерується і валідується з різним результатом через подвійне кодування
2. `/api/auth/me` потрапляє під `permitAll()` і не вимагає автентифікації явно

### Рішення

**Виправлення 1 — `JwtUtil.java`**: прибрати зайве кодування, використовувати ключ напряму:

```java
// Замість:
byte[] keyBytes = Decoders.BASE64.decode(
    java.util.Base64.getEncoder().encodeToString(secret.getBytes())
);

// Правильно — просто перетворити рядок у байти:
byte[] keyBytes = secret.getBytes(java.nio.charset.StandardCharsets.UTF_8);
```

**Виправлення 2 — `SecurityConfig.java`**: вивести `/api/auth/me` з-під `permitAll()`:

```java
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/auth/register", "/api/auth/login", "/api/auth/refresh").permitAll()
    // /api/auth/me — НЕ в permitAll, потрапляє під anyRequest().authenticated()
    .requestMatchers(HttpMethod.GET, "/api/texts", "/api/texts/**").permitAll()
    .requestMatchers("/swagger-ui/**", "/swagger-ui.html", "/api-docs/**").permitAll()
    .anyRequest().authenticated()
)
```

## Проблема 2 — Чи правильно що `/api/texts` є незахищеним?

### Аналіз

Поточний стан: `GET /api/texts` та `GET /api/texts/{id}` дозволені без токена.

Це **свідоме архітектурне рішення**, і воно **цілком коректне** для даного застосунку. Ось чому:

| Ресурс | Доцільність відкритого доступу |
|--------|-------------------------------|
| `GET /api/texts` | ✅ Тексти — публічний навчальний контент. Відкритий доступ дозволяє: показати preview до реєстрації, легко тестувати через Swagger/curl без токена |
| `POST /api/results` | 🔒 Результати — особисті дані, завжди захищені |
| `GET /api/results/me` | 🔒 Завжди захищений |

**Аналогія**: на Coursera можна переглянути опис курсу без реєстрації, але переглядати лекції — тільки після входу.

### Рекомендація

Залишити `GET /api/texts` відкритим — це **хороша практика** для публічного контенту. Якщо в майбутньому з'явиться платний або приватний контент, тоді закрити.

## Проблема 3 — З фронтенду не вдається увійти/зареєструватися

### Причина

Фронтенд (`Vite dev server`) запускається на порту **5173** за замовчуванням, бекенд — на **8080**.

**Перевірка `application.properties`**:
```
app.cors.allowed-origins=http://localhost:5173,http://localhost:3000
```
CORS налаштовано для `5173` та `3000`.

**Перевірка `api.ts`**:
```typescript
const BASE_URL = import.meta.env.VITE_API_URL ?? 'http://localhost:8080';
```
Якщо файл `.env` не підхопився або `VITE_API_URL` не визначено — фронтенд звертається до `localhost:8080`.

Найімовірніші причини проблеми:
1. **Файл `.env` не прочитаний** — Vite читає `.env` тільки при старті. Якщо `bun dev` запущено до того, як `.env` було створено — треба перезапустити.
2. **Бекенд не запущений або запущений на іншому порту** — треба перевірити на якому порту слухає Spring Boot.
3. **CORS помилка** — якщо фронтенд запущено на порту відмінному від `5173` і `3000`.

### Щодо пропозиції змінити порти (бекенд → 8000, фронтенд → 3000)

Це можливо, але **не обов'язково** — проблема не в номерах портів, а у тому що вони мають збігатися у CORS та `.env`. Якщо змінювати — треба оновити обидва місця.

**Рекомендація**: залишити стандартні порти (`8080` для Spring, `5173` для Vite) і просто виправити причину (перезапустити сервер після `.env`). Але якщо є бажання змінити — зміни мінімальні.

## Проблема 4 — `ACCESS_TOKEN_KEY` та `REFRESH_TOKEN_KEY` захардкоджені у фронтенді

### Причина непорозуміння

Це **не проблема** і не причина того, що авторизація не працює.

`ACCESS_TOKEN_KEY` та `REFRESH_TOKEN_KEY` — це **ключі localStorage** (назви "комірок" у браузері), а не самі токени. Вони визначають, **під якою назвою зберігати** токени локально:

```typescript
const ACCESS_TOKEN_KEY = 'rs_access_token';   // ← ім'я "полички" в localStorage
const REFRESH_TOKEN_KEY = 'rs_refresh_token'; // ← ім'я "полички" в localStorage
```

Самі **токени (JWT)** приходять з бекенду у відповіді на `/api/auth/login` або `/api/auth/register` і зберігаються під цими ключами:

```typescript
export function setTokens(accessToken: string, refreshToken: string): void {
  localStorage.setItem(ACCESS_TOKEN_KEY, accessToken); // зберігаємо токен що прийшов з бекенду
  localStorage.setItem(REFRESH_TOKEN_KEY, refreshToken);
}
```

Таким чином архітектура **правильна**: бекенд генерує токени → фронтенд зберігає їх у localStorage під фіксованими ключами. Це стандартна практика.
